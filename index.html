📋 一、概述

远程桌面软件的核心技术挑战在于：如何在有限带宽下实现低延迟、高画质的屏幕实时传输。

关键性能指标
  • 延迟：端到端延迟 <30ms 为优秀，<60ms 可接受
  • 帧率：60 FPS 为流畅标准
  • 带宽效率：100KB/s~500KB/s 应可完成办公任务
  • 画质：支持 4K、HDR、4:4:4 色度采样为高端

📸 三、屏幕捕获技术

3.1 各平台捕获方案

平台 捕获技术 特点
Windows DXGI Desktop Duplication GPU 直接输出帧，零拷贝，延迟最低
Windows Windows.Graphics.Capture UWP API，支持窗口级捕获
macOS ScreenCaptureKit Apple 原生 API，macOS 12.3+
Linux KMS/DRM 内核级直接访问显存
Linux X11 / Wayland 用户态捕获，兼容性好
Linux NvFBC NVIDIA 专有，帧缓冲直捕

3.2 变化检测算法

远程桌面不会每帧传输完整画面，而是只传输变化区域：

算法 原理 适用场景
脏矩形检测 (Dirty Rectangle) 比较前后帧，标记变化的矩形区域 VNC、早期远程桌面
帧差分 (Frame Difference) 像素级 XOR 比较，提取差异区域 文本/静态界面场景
GPU 硬件检测 DXGI 直接返回脏区域列表 Windows 现代方案
分块哈希 将屏幕分块计算哈希，变化块重传 带宽受限场景

🏢 五、主流软件技术分析

🇨🇳 ToDesk

闭源 国产
  • 传输引擎：自研 ZeroSync 引擎
  • 编码技术：H.265 + AV1 双编码
  • 色彩支持：4:4:4 色深真彩
  • 硬件加速：NVENC / AMD VCE
  • 协议：UDP + TCP 智能切换
  • 延迟：局域网 3ms，跨省 18-40ms
  • 特色：P2P 穿透率 97%，支持 8K/360FPS

🌻 向日葵

闭源 国产
  • 传输协议：自研 Sunlogin 协议
  • 基础：基于 RDP 扩展
  • 编码：自研高效编码算法
  • 带宽适应：智能调节，弱网 720P@30fps
  • 多通道：画面+文件+语音同传
  • 分辨率：最高 4K
  • 特色：硬件生态（开机棒/插座）

🇩🇪 AnyDesk

闭源 德国
  • 核心编码器：DeskRT（专为 GUI 设计）
  • 帧率：稳定 60 FPS
  • 延迟：局域网 <16ms
  • 带宽效率：100KB/s 即可办公
  • 加密：TLS 1.2 + RSA 2048
  • 全球节点：67 个边缘服务器
  • 特色：跨洋延迟可低至 30ms

🦀 RustDesk

开源 ⭐ 114k
  • 语言：Rust + Flutter
  • 屏幕捕获：scrap 库（自研）
  • 编码器：VP8/VP9, H.264/H.265
  • 硬件加速：hwcodec 模块
  • 协议：自研 + NAT 穿透
  • 部署：支持自建服务器
  • GitHub：rustdesk/rustdesk

☀️ Sunshine

开源 ⭐ 26k
  • 定位：游戏串流服务器
  • 客户端：Moonlight 全家桶
  • 编码支持：NVENC/AMF/QuickSync/VAAPI
  • 捕获：DXGI/KMS/ScreenCaptureKit
  • 延迟目标：端到端 <30ms
  • HDR：支持 HDR 串流
  • GitHub：LizardByte/Sunshine

🖥️ VNC (TightVNC)

开源 经典
  • 协议：RFB (RFC 6143)
  • 编码：Tight/ZRLE/Hextile
  • 特点：协议简单，跨平台
  • 延迟：相对较高
  • 适用：服务器管理、轻量远程
  • 扩展：TurboVNC 支持 H.264
  • GitHub：TigerVNC/tigervnc

DeskRT 编码器技术推断

AnyDesk DeskRT 是专为桌面 GUI 设计的编码器（非通用视频编码），根据公开资料推断其技术特点：

  • 针对桌面特性优化：大面积纯色、文字边缘锐利、窗口移动
  • 区分静态区域和动态区域，采用不同压缩策略
  • 对重复图案（图标、按钮）进行字典编码
  • 动态内容可能回退到类 H.264 编码

📜 六、VNC/RFB 协议（学术基础）

VNC 的 RFB (Remote Framebuffer) 协议是远程桌面的学术基础，定义于 RFC 6143。

6.1 标准编码类型

编码 ID 原理 适用场景
Raw 0 原始像素数据 局域网/基准测试
CopyRect 1 引用已有矩形区域 窗口移动/滚动
RRE 2 矩形分解为纯色子块 简单图形界面
Hextile 5 16x16 分块 + 子编码 通用场景
ZRLE 16 Zlib + Run-Length 编码 高压缩率需求
Tight 7 JPEG + Zlib + 调色板混合 TightVNC 高效扩展

6.2 Tight 编码流程

输入矩形 → 分析内容类型 ├── 纯色/渐变 → 调色板编码 + Zlib ├── 复杂图像 → JPEG 压缩 └── 文字/线条 → 基础过滤 + Zlib

6.3 现代扩展

扩展 来源 特性
H.264 Encoding TurboVNC 硬件加速 H.264 编码
Tight PNG noVNC PNG 压缩替代 JPEG
QEMU VMware QEMU 针对虚拟机优化

核心结论

  1. 商业软件（ToDesk、AnyDesk）的核心竞争力在于自研编码器和传输协议优化，技术细节不公开
  2. 开源方案（RustDesk、Sunshine）提供完整技术栈，适合学习和二次开发
  3. 屏幕传输的关键优化点：
    • 高效屏幕捕获（DXGI/ScreenCaptureKit）
    • 硬件编码加速（NVENC/AMF/QuickSync）
    • 智能变化检测（脏矩形/GPU 硬件检测）
    • 自适应传输协议（UDP+TCP 混合、P2P 穿透）
  4. 学术研究可参考 RFC 6143 (RFB) 和 HEVC Screen Content Coding 相关论文
